Data Division

All of the data we use in our COBOL program gets set up in the DATA DIVISION, and we use a data relationship to add structure to it.

MOVE and COMPUTE

These are two reserved words we use to manipulate the values of variables. MOVE moves data from one place into a variable. COMPUTE is used to do mathematical operations.

File Handling

This would involve the SELECT and ASSIGN clauses in the ENVIRONMENT DIVISION, FD statements in the DATA DIVISION, and in the PROCEDURE DIVISION, we'd be using the OPEN, CLOSE, READ-INTO and WRITE-FROM statements

The ENVIRONMENT and DATA divisions are where inputs and outputs are described, which in turn are used in the PROCEDURE DIVISION

Since we're working with z/os, the datasets we'll be using will be of the sequential data storage type. A dataset in z/os consists of records. Each record is a line in a dataset, and has a defined length. In COBOL, data can be loaded as a block for efficiency, where a block is a group of records, and put in a buffer.

Procedure Division

This is where we open the files, specifically in the OPEN-FILES section. We close files in the CLOSE-STOP section at the end of the program, or when we're done using the file.

COBOL is often used to process lots of data. As such, we'd need some kind of for-loop to go through RECORDs. In COBOL, the reserved word for that is PERFORM, and also the combined PERFORM UNTIL or PERFORM THRU.

Program Structure

COBOL is often described as a structured programming language, meaning that it relies on the concept of passing control and following a flow structure as the program executes. The way you write your code spells out how it should be executed. You use the structure of COBOL to make sure your code flows the way it ought to.A lot of the code you write in COBOL needs to be transparent, readable and easily diagnosed.

There is some ability to jump around a little bit in the code, but be sure to use that sparingly. Don't make it the backbone of your code. COBOL passes control mainly using the PARAGRAPHS and the PERFORM keyword.

A note about GOTO: Don't use it unless you really need to do it and know for sure what it's going to do exactly. In contrast with PERFORM, GOTO will go to where you direct it in the code and start executing from there. Take care using GOTO because you might be losing some of that structure we talked about previously.

Paragraphs

When you begin planning a program, start thinking in paragraphs. Paragraphs are defined in the procedure division and start from column 8. You can name them anything you like other than a reserved keyword. The paragraph can contain one or more sentences, and you terminate the declaration/definition of a paragraph with a period.

Spend some time in the beginning planning the flow and structure of your program, e.g. which part will be handled in which paragraph.This helps with debugging errors and serves as a baseline reference you can check against.

If you find yourself repeating the same code, that's probably a good place for to use a paragraph. Put that code into a paragraph, and give it a name. Also spend some time thinking up a good name for paragraphs, ideally names that give you a good a idea of what the paragraph is supposed to be doing. Loops are also something that belongs nicely in PARAGRAPHS. Put them in there using the PERFORM keyword. A nice simple way to set up loops is with the keyword TIMES.

A paragraph can come before or after the code that calls it, but you should order your paragraphs in the order that they'd be executed in, or as best as you can try anyway.
